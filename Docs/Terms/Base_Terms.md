# Base_Terms

## Назначение
Этот файл содержит минимальный стартовый пакет терминов продуктового репозитория.
Полный словарь `BytePress` сюда не копируется.

## Индекс
- TERM-000019 — Каркас репозитория
- TERM-000020 — Текущая истина
- TERM-000021 — Рабочая ветка
- TERM-000007 — Дорожная карта
- TERM-000008 — Реестр работ
- TERM-000009 — План

## Стартовый пакет терминов

### TERM-000019 — Каркас репозитория
Определение: Минимальный управляемый каркас репозитория продукта, который bootstrap materialize по умолчанию для раннего product-start: корневые точки входа, ограниченный `Docs/*`, `Plans/*`, `Logs/*`, лёгкий `Pipeline/*`, локальный `Tools/*`, bounded `Templates/*`, bounded `Schemas/*` и переходный `scripts/*` слой только как оболочки к `Tools/*`.
Зачем нужен на старте: Объясняет, какой минимальный каркас bootstrap создаёт по умолчанию.

### TERM-000020 — Текущая истина
Определение: Актуальный набор подтверждённых ответов о продукте или системе, которым в текущем scope владеет owner-document `Docs/Discovery/Interview.md`.
Зачем нужен на старте: Удерживает аналитический гейт до явных ответов пользователя.

### TERM-000021 — Рабочая ветка
Определение: Task-ветка формата `<type>/<NNNNNN>-<slug>`, из которой начинается любой writable pass и через которую проходит локальная проверка, push и PR-путь.
Зачем нужен на старте: Фиксирует, что проход с правками начинается только из task-ветки.

### TERM-000007 — Дорожная карта
Определение: Средне- и долгосрочная карта этапов (`stage`), целей и зависимостей продукта или системы, которая владеет статусами этапов и не распадается на список отдельных проходов.
Зачем нужен на старте: Помогает читать планирование этапов продукта.

### TERM-000008 — Реестр работ
Определение: Упорядоченный список задач (`task`) текущего этапа с типом, приоритетом, статусом и связями; backlog владеет статусами задач активного этапа, а дальний горизонт остаётся в roadmap.
Зачем нужен на старте: Помогает читать планирование задач текущего этапа.

### TERM-000009 — План
Определение: Формальный план одного прохода (`pass`) конкретного изменения, строящийся на основе backlog-задачи и связанных требований и решений; `Plan` не владеет глобальным статусом этапа, а runtime-контекст не подменяет его как канонический артефакт.
Зачем нужен на старте: Помогает читать и вести один текущий pass продукта.
