# Interview

## Назначение интервью

`Docs/Discovery/Interview.md` является owner-document текущей аналитической истины о продукте до её перевода в требования и planning contour.

---

## Правило актуальности

Интервью хранит только актуальные вопросы и ответы. История изменений фиксируется через `Plans/*` и `Logs/*`.

---

## Статус текущей истины

Статус_текущей_истины: Не_подтверждена
Правило: bootstrap placeholders и пустые ответы не считаются подтверждением current truth.
Правило: первый writable action по этому интервью допускается только после открытия task-ветки.
Правило: до явных ответов пользователя это интервью не разрешает изменения вне `Docs/Discovery/*`, `Plans/*` и `Logs/*`.

---

## Правило delta-интервью

Если для подтверждения current truth достаточно уточнить только часть вопросов, допускается узкое delta-интервью вместо полного повторного интервью. Оно всё равно обязано сохранять тот же format contract: нумерованные вопросы, буквенные варианты там, где выбор ограничен, и рекомендуемый вариант там, где это уместно.

---

## Вопросы и ответы

### 1. Какую текущую истину о продукте нужно зафиксировать первой?
Ответ: Не подтверждено пользователем.

Варианты ответа:
- A. Основной пользователь и его задача.
- B. Граница первой версии продукта.
- C. Ключевое ограничение реализации.

Рекомендуемый вариант: A — он быстрее всего даёт опорную current truth для следующего предметного pass.

### 2. Какая граница первой версии сейчас выглядит самой вероятной?
Ответ: Не подтверждено пользователем.

Варианты ответа:
- A. Один пользовательский сценарий и один основной результат.
- B. Полный release-ready продукт сразу.
- C. Только технический эксперимент без user value.

Рекомендуемый вариант: A — он удерживает первую версию в узком и проверяемом scope.

### 3. Какой факт о продукте пока остаётся самым неопределённым?
Ответ: Не подтверждено пользователем.

### 4. Какое ограничение сильнее всего влияет на первый pass продукта?
Ответ: Не подтверждено пользователем.

Варианты ответа:
- A. Ограничение времени и узкого scope.
- B. Обязательная полная автоматизация с первого дня.
- C. Требование открыть сразу все будущие домены.

Рекомендуемый вариант: A — ранний product-start contour должен оставаться узким и управляемым.

### 5. Какой основной пользовательский результат должен стать наблюдаемым после первого pass?
Ответ: Не подтверждено пользователем.

### 6. Какие артефакты являются первыми owner-documents продукта для этого scope?
Ответ: `Docs/Discovery/Interview.md`, `Docs/Product/*`, `Plans/*` и при необходимости `Docs/Technical/*`.

### 7. Какой route дальнейшего уточнения discovery сейчас нужен после интервью?
Ответ: Не подтверждено пользователем.

Варианты ответа:
- A. Открыть отдельный pass на `Discussion` / `Research` / `Requirements`, если minimum уже недостаточен.
- B. Сразу переписать планы без фиксации новой current truth.
- C. Оставить discovery только в устном виде.

Рекомендуемый вариант: A — он открывает расширенный discovery contour только по явному решению.

### 8. Какие артефакты нужно проверить после обновления этого интервью?
Ответ: `Plans/Roadmap.md`, `Plans/Backlog.md`, active `Plans/*`, `Logs/ChangeLog.md`, `Logs/QualityLog.md`.
