# Workflows

## Основной путь продукта
1. Назвать фазу, рабочий поток и гейт.
2. Открыть task-ветку до первого записываемого действия.
3. Подтвердить `ROAD/BACK/PLAN`.
4. Прочитать документы-владельцы по области прохода.
5. Зафиксировать противоречия, гипотезы и блокирующие вопросы.
6. Внести минимальные изменения.
7. Выполнить проверки нужного уровня.
8. Закрыть `Logs/*`, сделать смысловые коммиты, один push и PR через `gh`.

## Первый старт продукта
Фаза: `Подтверждение текущей истины`.
Рабочий поток: `Первый старт продукта`.
Гейт: `Гейт текущей истины`.

1. Не считать начальный запрос пользователя полным подтверждением текущей истины.
2. Не заполнять `Docs/Discovery/Interview.md` догадками агента.
3. Фиксировать неявные предположения отдельно как гипотезы.
4. Задать пользователю блокирующие вопросы.
5. Не записывать ограничения первого прохода как подтверждённые требования без явного источника.
6. Не записывать стек и зависимости как подтверждённые требования без явного источника.
7. После явных ответов синхронизировать `Plans/*` и `Logs/*`.
8. Запустить `python3 Tools/product_check.py --repo . --mode auto`.

## Предметный проход
Фаза: `Реализация`.
Рабочий поток: `Предметный проход`.
Гейт: `Гейт реализации`.

1. Проверить, что текущая истина подтверждена.
2. Подтвердить ограничения первого прохода.
3. Подтвердить источник выбора стека: ответ пользователя, профиль продукта, требование или техническое решение.
4. Разделить документацию, код, тесты, проверки и журнальное закрытие.
5. Если запуск нельзя проверить, записать `не проверено`.
6. Не открывать новые домены без владельца, потребителя и проверки.

## Уровни проверок
- `Структура` — обязательные файлы и запрет удалённых доменов.
- `Тесты` — автоматические тесты кода.
- `Запуск` — фактический CLI, service или smoke route.
- `GUI-запуск` — доступность GUI-модуля и среды.
- `Ручная проверка` — явно записанное действие, которое нельзя честно автоматизировать.

## Журнальное закрытие
- `ChangeLog.md` фиксирует факт изменения.
- `QualityLog.md` фиксирует выполненные проверки и непроверенные зоны.
- `ADRlog.md` обновляется для архитектурных, процессных и продуктово-договорных решений.

## PR-маршрут
1. Выполнить одну финальную отправку рабочей ветки.
2. Проверить через `gh`, что для рабочей ветки нет открытого PR.
3. Создать PR через `gh pr create`.
4. GitHub connector не использовать для создания PR.
5. Если `gh` недоступен, выполнить push и вернуть параметры для ручного PR.
